DATABASE TRIGGERS ARE DATABASE OBJECTS CREATED ON THE CLIENT AND STORED ON THE SERVER IN THE DBMS SYSTEM TABLE. THESE DATABASE OBJECTS CONSISTS OF THE FOLLOWING DISTINCT SECTIONS :
1) A NAMED DATABASE EVENT
2) A PL/SQL BLOCK THAT WILL EXECUTE WHEN THE EVENT OCCURS

THE OCCURENCE OF THE DATABASE EVENT IS STRONGLLY BOUND TO TABLE DATA BEING CHANGED.

A DATABASE  TRIGGER IS A KIND OF STORED PROCRDURE THAT CAN BE ASSOCIATED WITH A SPECIFIC DATABASE TABLE DEFINING AN ACTION TO BE EXCUTED IMPLICITLY ( I.E. EXCUTED BY THE DATABASE ENGINE ITSELF), WHEN AN INSERT, UPDATE OR DELETE IS ISSUED AGAINST A TABLE THORUGH AN APPILCATION. THE MAJOR ISSUES THAT MAKE THESE TRIGGERS STANDALONE ARE THAT, THEY ARE FIRED IMPILICTLY (I.E)INTERNALLY BY THE DATABASE ENGINE ITSELF AND NOT EXPILICTLY THAT IS CALLED BY THE USER.

IMPORTANCE OF TRIGGERS :
1) TRIGGERS VERIFY DATA INTEGRITY WHEN IT IS INSERTED OR UPDATED
2) TRIGGERS ENSURE THAT BUSINESS RULES RELATED TO A TABLE ARE PERFORMED WHEN RECORDS ARE INSERTED, UPDATED OR DELETED
3) TRIGGERS IMPLEMENT CASCADING DELETE 
4) TRIGGERS AUTOMATICALLY GENERATE DERIVED COLUMN VALUES
5) TRIGGERS ENFORCE SECURITY RULES
TRIGGERS CAN BE USED TO SUPPLEMENT REFERENCIAL INTEGRITY(THOUGH THEY SHOULD NOT BE USED TO REPLACE THEM)

USE OF DATABASE TRIGGERS :
SINCE THE ORACLE ENGINE SUPPORTS DATABASE TRIGGERS IT PROVIDES A HIGHLY CUSTOMIZABLE DATABASE SYSTEM SOME OF THE USES TO WHICH THE DATABASE TRIGGERS CAN BE PUT TO CUSTOMIZE MANAGEMENT INFORMATION BY THE ORACLE ENGINE ARE AS FOLLOWS :
1) A TRIGGER CAN PERMIT DML STATEMENTS AGAINST A TABLE ONLY IF THEY ARE ISSUED DURING REGULAR BUSINESS HOURS OR ON PREDTERMINED WEEK-DAYS 
2) A TRIGGER CAN ALSO BE USED FOR A AUDIT TRAIL OF A TABLE I.E. TO STORE OR MODIFY AND DELETED RECORDS OF THE TABLE ALONG WITH THE OPERATION PERFORMED AND THE TIME ON WHICH THE OPERATION WAS PERFORMED 
3) IT CAN BE USED TO PREVENT INVALID TRANSACTIONS
4) IT CAN BE USED TO ENFORCE COMPLEX SECURITY AUTHORIZATIONS.

TYPES OF TRIGGERS :
WHILE DEFINING A TRIGGER, NO. OF TIMES THE TRIGGER ACTION IS TO BE EXECUTED CAN BE SPECIFIED. IN THAT CASE THE TRIGGERS ARE GENERALLY CAN BE OF 2 TYPES :
1) ROLLABLE TIGGER : EXECUTES ONCE FOR EVERY ROW IN A TRANSACTION. A ROLLABLE TRIGGER IS EXECUTED EVERYTIME THE TABLE IS AFFECTED BY THE TRIGGERING STATEMENT. FOR EXAMPLE : IF AN UPDATE STATEMENT AFFECTS 10 ROWS IN A TABLE, THE TRIGGER IS FIRED 10 TIMES. A ROLLABLE TRIGGER IS IDENTIFIED BY THE "FOR EACH ROW" CLAUSE IN TRIGGER COMMAND.
2) STATEMENT LEVEL TRIGGER : EXECUTES ONLY ONCE FOR EACH TRANSACTION OR TRIGGERING STATEMENT, NO MATTER HOW MANY ROWS IT AFFECTS. FOR EXAMPLE : IF AN UPDATE STATEMENT IS USED AND IT AFFECTS 10 ROWS IN A TABLE, THE TRIGGER IS FIRED ONLY ONCE. THIS TYPE OF TRIGGERS IS THE DEFAULT TYPE OF TRIGGERS. THESE ARE IDENTIFIED BY COMMITING THE FOR EACH ROW CLAUSE IN THE TRIGGER COMMAND.

TRIGGER TIMING : 
WHEN TRIGGERS ARE DEFINED, IT NEEDS TO BE SPECIFIED WHEN THE TRIGGERS SHOULD BE FIRED. TRIGGERS CAN BE EXECUTED IMMEDIATELY BEFORE OR AFTER INSERT OR UPDATE OR DELETE.

1) BEFORE TRIGGERS : THE BEFORE TRIGGERS EXECUTE THE TRIGGER ACTION BEFORE THE TRIGGERING STATEMENT. THIS TYPES OF TRIGGERS ARE COMMONLY USED IN THE FOLLOWING SITUATIONS :

i) BEFORE TRIGGERS ARE USED WHEN THE TRIGGER ACTION SHOULD DETERMINE WHETHER OTR NOT THE TRIGGERING STATEMENT SHOULD BE ALLOWED TO COMPLETE. BY USING THE BEFORE TRIGGER, USER CAN ELEMINATE UNNECESSARY TRIGGERING STATEMENT.

ii) BEFORE TRIGGERS ARE USED TO DERIVE THE SPECIFIC COLUMN VALUES BEFORE COMPLETING A TRIGGERING INSERT OR UPDATE STATEMENT

2) AFTER TRIGGER : IT EXECUTES THE TRIGGER ACTION AFTER THE TRIGGERING STATEMENT IS EXECUTED. THESE TYPE OF TRIGGERS ARE COMMONLY USED IN THE FOLLOWING SITUATIONS : 
i) AFTER TRIGGERS ARE USED WHEN THE TRIGGERING STATEMENT SHOULD BE COMPLETE  BEFORE EXECUTING THE TRIGGER ACTION.
ii) IF A BEFORE TRIGGER IS ALRRADY PRESENT AND AFTER TRIGGER CAN PERFORM DIFFERENT ACTIONS ON SAME TRIGGERING STATEMENT.

USING BEFORE AND AFTER CLAUSE FOLLOWING 12 TRIGGERS CAN BE CREATED ON A TABLE :
TRIGGER	:                                FUNCTION :
1) BFEORE INSERT STATEMENT LEVEL	1) FIRES ONCE BEFORE AN INSERT TRANSACTION
2) BEFORE INSERT ROW LEVEL	        2) FIRES BEFORE EACH NEW RECORD IS CREATED
3) BEFORE INSERT STATEMENT LEVEL	3) FIRES ONCE AFTER AN INSERT TRANSACTION
4) AFTER INSERT ROW LEVEL	        4) FIRES AFTER EACH NEW RECORD IS CREATED
5) BEFORE UPDATE STATEMENT LEVEL	5) FIRES ONCE BEFORE AN UPDATE TRANSACTION
6) BEFORE UPDATE ROW LEVEL	        6) FIRES BEFORE EACH RECORD IS UPDATED
7) AFTER UPDATE STATEMENT LEVEL	        7) FIRES ONCE AFTER AN UPDATE TRANSACTION
8) AFTER UPDATE ROW LEVEL	        8) FIRES AFTER EACH RECORD IS UPDATED
9) BEFORE DELETE STATEMENT LEVEL	9) FIRES ONCE BEFORE A DELETE TRANSACTION
10) BEFORE DELETE ROW LEVEL	        10) FIRES BEFORE EACH RECORD IS DELETED 
11) AFTER DELETE STATEMENT LEVEL	11) FIRES ONCE AFTER A DELETE TRANSACTION
12) AFTER DELETE ROW LEVEL	        12) FIRES AFTER EACH RECORD IS DELETED

	INSERT INTO CUSTOMER VALUES(1001,'ROBIN','KOLKATA','SAVINGS'); - EXPLICITLY
2)	ACCBALANCE (ACCNO, BALANCE)
	INSERT INTO ACCBALANCE VALUES(1001,0); - IMPLICITLY BY TRIGGER
3)	TRANSACTION (ACCNO, TRANDATE, TRANTYPE, AMOUNT)

SET SERVEROUTPUT ON
CREATE OR REPLACE TRIGGER TRIG01 AFTER INSERT ON CUSTOMER FOR EACH ROW 
BEGIN
	INSERT INTO ACCBALANCE VALUES(:NEW.ACCNO,0);
	DBMS_OUTPUT.PUT_LINE('ACCOUNT BALANCE UPDATED SUCCESSFULLY');
END;

TRIGGERING EVENT/STATEMENT:
---------------------------
SQL> INSERT INTO CUSTOMER VALUES(1001,'ROBIN','KOLKATA','SAVINGS');
OUTPUT: ACCOUNT BALANCE UPDATED SUCCESSFULLY


VALUES  OF: NEW AND: OLD CORRELATION NAMES:
------------------------------------------
ON THE BASIS OF THE OPERATION PERFORMED, THE OLD AND NEW VALUES HAS THE FOLLOWING MEANING:
1) A DATABASE TRIGGER FIRED BY AN INSERT STATEMENT CAN ACCESS ONLY THE NEW COLUMN VALUES FOR AN AFTER TRIGGER, THE ROW IS CREATED BY INSERT, THEREFORE OLD VALUES ARE NULL
2) A DATABASE TRIGGER FIRED BY A DELETE STATEMENT CAN ACCESS ONLY THE OLD COLUMN VALUES FOR BOTH BEFORE AND AFTER TRIGGER. AFTER THE ROW IS DELETED THE NEW VALUES ARE NULL.
3) A DATABASE TRIGGER FIRED BY AN UPDATE STATEMENT CAN ACCESS BOTH THE OLD AND NEW COLUMN VALUES FOR BOTH BEFORE AND AFTER TRIGGER.
VALUES OF :NEW AND :OLD CORRELATION NAMES:
----------------------------------------------------------------------------------
									INSERT
					|------------------+-------------------|
				BEFORE									AFTER
		|---------+---------|					|---------+---------|
	OLD(NULL)			NEW(VALUE)			OLD(NULL)			NEW(VALUE)
----------------------------------------------------------------------------------

									DELETE
					|------------------+-------------------|
				BEFORE									AFTER
		|---------+---------|					|---------+---------|
	OLD(VALUE)			NEW(NULL)			OLD(VALUE)			NEW(NULL)
----------------------------------------------------------------------------------

									UPDATE
					|------------------+-------------------|
				BEFORE									AFTER
		|---------+---------|					|---------+---------|
	OLD(OLD VALUE)		NEW(NEW VALUE)		OLD(OLD VALUE)		NEW(NEW VALUE)
----------------------------------------------------------------------------------
EXAMPLE : 
CONSIDERING THE TABLES CUSTOMER ACCOUNT BALANCE AND CUTOMER, CREATE A TRIGGER ON CUSTOMER 
SUCH THAT WHEN A NEW RECORD IS INSERTED INTO THE CUSTOMER TABLE AND ENTRY WILL AUTOMATICALLY
BE MADE INTO THE ACCBALANCE TABLE WITH THE ACCNO SPECIFIED AND THE BALANCE AMOUNT AS RS 1000 
IF ACCTYPE SAVINGS, RS 5000 IF ACCTYPE IS CURRENT AND RS 10000 IF ACCTYPE IS FIXED.
IF THE TRANSACTION IS MADE SUCCESFULLY A MESSAGE WILL BE DISPLAYED AS BALANCE UPDATED SUCCESFULLY.

 ANSWER :
 CLEAR SCREEN
SET SERVEROUTPUT ON
CREATE OR REPLACE TRIGGER TRIG01 AFTER INSERT ON CUSTOMER FOR EACH ROW 
DECLARE 
	AMT ACCBALANCE.BALANCE%TYPE;
BEGIN
	IF UPPER (:NEW.ACCTYPE) = 'SAVINGS' THEN
	AMT := 1000;
	ELSEIF UPPER(:NEW.ACCTYPE) = 'CURRENT' THEN
	AMT := 5000;
	ELSEIF UPPER(:NEW.ACCTYPE) = 'FIXED' THEN
	AMT := 10000;
	END IF;
	INSERT INTO ACCBALANCE VALUES(:NEW.ACCNO,0);
	DBMS_OUTPUT.PUT_LINE('ACCOUNT BALANCE UPDATED SUCCESSFULLY');
END;
/
